

# Node.js文件操作中的竞态条件问题

## 核心问题：时间差 (Time-of-Check to Time-of-Use)

竞态条件的本质是：**检查的那一刻**和**使用的那一刻**之间，时间流逝了。在这微小的时间窗口中，文件系统的状态可能已经发生了变化。

## fs.exists 的竞态条件

### 问题描述

`fs.exists` 存在明显的竞态条件风险，这是其被官方废弃的主要原因。

### 执行流程

1. **检查阶段**：调用 `fs.exists('file.txt')`，系统返回 `true`（文件存在）
2. **时间窗口**：在检查结果返回后，实际操作执行前，另一个进程删除了 `file.txt`
3. **操作阶段**：执行 `fs.readFile('file.txt')`
4. **结果**：抛出 `ENOENT` 错误（文件不存在），程序基于过期信息做出错误决策

### 代码示例

```
// ❌ 不安全的写法
fs.exists('file.txt', (exists) => {
  if (exists) {
    // 时间窗口：文件可能在此时被删除
    fs.readFile('file.txt', (err, data) => {
      // 可能抛出 ENOENT 错误
    });
  }
});
```

## fs.stat 的竞态条件

### 问题描述

虽然 `fs.stat` 比 `fs.exists` 提供更多信息，但在"先检查后使用"的模式下，同样存在竞态条件。

### 执行流程

1. **检查阶段**：调用 `fs.stat('data/config.json')`
2. **系统返回**：文件存在，是普通文件，大小100字节
3. **时间窗口**：另一个进程删除 `config.json` 并创建同名目录
4. **操作阶段**：基于检查结果执行 `fs.readFile('data/config.json')`
5. **结果**：抛出 `EISDIR` 错误（试图读取目录）

### 代码示例

```
// ❌ 同样存在风险
fs.stat('data/config.json', (err, stats) => {
  if (!err && stats.isFile()) {
    // 时间窗口：文件状态可能在此时改变
    fs.readFile('data/config.json', (err, data) => {
      // 可能抛出 EISDIR 错误
    });
  }
});
```

## 对比分析

| 特性 | fs.exists | fs.stat | 直接操作 |
| ------ |------ |------ |------ |
| 竞态条件风险 | 高 | 中 | 低 |
| 信息量 | 仅存在性 | 完整元数据 | 操作结果 |
| 错误处理 | 不规范 | 规范 | 最佳 |
| 推荐程度 | ❌ 已废弃 | ⚠️ 慎用 | ✅ 推荐 |

## 真正的最佳实践

### 原子性操作原则

**不要问"能不能做"，直接"去做"，然后处理结果。**

### 推荐模式：Eager Open

```
// ✅ 推荐的写法
fs.readFile('file.txt', (err, data) => {
  if (err) {
    if (err.code === 'ENOENT') {
      console.log('文件不存在，执行创建逻辑');
    } else if (err.code === 'EISDIR') {
      console.log('路径是目录，不能读取');
    } else {
      console.log('其他错误:', err);
    }
  } else {
    console.log('文件读取成功');
  }
});
```

## 总结

1. `fs.exists`：存在竞态条件，设计不规范，**绝对不要使用**
2. `fs.stat`：虽然设计规范且信息丰富，但在"先检查后使用"模式下**依然存在竞态条件**
3. **直接操作**：**最安全**的方案，通过捕获具体错误码来处理各种情况
4. **核心原则**：让操作系统在操作瞬间判断状态，消除"检查-使用"之间的时间差

通过直接操作文件并捕获特定错误码的方式，可以构建出最健壮、最安全的文件处理逻辑。
