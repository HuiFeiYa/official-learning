🧪 前端内存泄漏实验室

操作步骤：点击“触发泄漏” -> 点击多次 -> 点击“强制 GC” -> 拍摄快照 -> 分析未释放的对象。

1. 分离的 DOM 节点 (Detached DOM) 0

原理：创建了大量 DOM 元素，移除出文档树后，仍然用全局数组持有它们的引用。
现象：Heap Snapshot 中出现大量 Detached HTMLDivElement。
// 全局变量意外持有了已移除的 DOM const detachedNodes = []; function createLeak() { const div = document.createElement('div'); div.innerHTML = 'I am a leaked node ' + Math.random(); // 模拟复杂结构 for(let i=0; i<100; i++) { div.appendChild(document.createElement('span')); } document.body.appendChild(div); // 移除 DOM document.body.removeChild(div); // ❌ 错误：虽然移除了，但数组还引用着它！ detachedNodes.push(div); } function cleanLeak() { // ✅ 正确：清空数组引用 detachedNodes.length = 0; }
状态：正常

2. 遗忘的定时器 (Timers) 0

原理：创建了 setInterval，组件/逻辑销毁时未清除。回调函数闭包持有了大对象。
现象：即使页面不动，内存也在涨。Snapshot 中看到 Closure 和 Array。
let timers = []; const bigData = new Array(10000).fill('Memory Hog'); function createLeak() { const id = setInterval(() => { // ❌ 闭包持有了 bigData，且定时器永不停止 console.log('Running...', bigData.length); }, 1000); timers.push(id); } function cleanLeak() { // ✅ 正确：清除所有定时器 timers.forEach(clearInterval); timers = []; }
状态：正常

3. 未清理的事件监听器 (Event Listeners) 0

原理：给 window 或 document 绑定事件，回调函数引用了局部大对象，且未解绑。
现象：Snapshot 中看到 Window 指向大量的 Closure。
let listeners = []; function createLeak() { const hugeObject = new Array(5000).fill({ data: 'Leak' }); const handler = () => { // ❌ handler 闭包持有了 hugeObject // 只要 window 还在，handler 就在，hugeObject 就无法回收 return hugeObject.length; }; window.addEventListener('resize', handler); listeners.push(handler); // 只是为了计数，实际泄漏是因为 addEventListener } function cleanLeak() { // ❌ 常见错误：只清空数组，没调用 removeEventListener! // listeners = []; // ✅ 正确做法需要保存 handler 引用并移除 (这里演示正确做法) // 注意：为了演示泄漏，上面的 createLeak 没有保存匿名函数的引用供移除 // 这里的 clean 实际上无法完全清理上面的匿名函数，除非我们在 createLeak 时保存了引用 // *修正演示*：我们将使用可移除的方式演示“忘记移除”的后果 alert("提示：此场景演示的是‘忘记调用 removeEventListener’。\n由于上面创建的是匿名函数且未保存引用，即使点击清理也无法移除旧的监听器（这就是泄漏！）。\n请点击触发多次，然后看 Snapshot 中的 Window Retainers。"); }
状态：正常

4. 意外的闭包引用 (Heavy Closures) 0

原理：在一个大作用域内定义小函数，并将小函数暴露给全局。小函数只用了小变量，但闭包锁住了整个大作用域。
现象：一个小的 Function 对象，Retainers 显示它持有一个巨大的 Array。
const globalCallbacks = []; function createLeak() { // 模拟一个巨大的局部数据 const massiveData = new Array(20000).fill('Secret Data'); // 定义一个小回调，只用到了 count let count = 0; const smallCallback = () => { count++; return count; }; // ❌ 错误：将回调放入全局数组 // 虽然 smallCallback 只用 count，但它所在的闭包环境持有了 massiveData globalCallbacks.push(smallCallback); } function cleanLeak() { // ✅ 正确：清空全局引用 globalCallbacks.length = 0; }
状态：正常

5. 无限增长的全局缓存 (Unbounded Cache) 0

原理：使用全局 Map/Object 做缓存，只存不删，或者键值对生成策略有问题（如用时间戳做 Key）。
现象：全局 Map 或 Object 体积无限膨胀。
const cache = new Map(); function createLeak() { // 模拟生成唯一的 Key (如时间戳 + 随机数) const key = Date.now() + '-' + Math.random(); const value = new Array(5000).fill('Cached Item'); // ❌ 错误：永远不清理旧缓存 cache.set(key, value); } function cleanLeak() { // ✅ 正确：限制缓存大小或清空 cache.clear(); // 或者实现 LRU 算法 }
状态：正常

注意：需要在 DevTools Settings -> Preferences -> Enable "Show heap snapshot object allocation" 并确保能访问 performance 接口。
如果按钮无效，请手动点击 DevTools Memory 面板的垃圾桶图标。
